Section 1
Vision & Philosophy
What a CPU-first engine actually means — and why the constraints are the point.

What "CPU-first" actually means

A CPU-first engine treats the processor as the primary rendering and simulation unit — not a command buffer dispenser for the GPU. The CPU owns the framebuffer, performs rasterization, runs the game simulation, manages all memory, and drives output. GPU involvement is optional and supplemental: for blitting the final framebuffer to screen, or minimal acceleration on integrated graphics.

This is fundamentally different from a "CPU-assisted GPU renderer." In GPU-first engines (every major modern engine), the CPU feeds draw calls and shader parameters to the GPU. In a CPU-first engine, the CPU is the renderer. The decision isn't about which is faster — it's about architectural ownership.

Core philosophical pillars

Constraint-driven design

Every architectural decision is forced by real hardware constraints. If a target machine has 2 GB RAM and an Intel HD 4000, that constraint shapes every subsystem. Constraints prevent feature creep and generate elegant solutions. PICO-8's 128×128 resolution and 16-color palette are not limitations — they are the design. Your potato target IS the spec.

Performance by architecture

True performance doesn't come from micro-optimizing bad architecture — it comes from designing systems where the fast path is the default path. Data flows linearly through cache. Allocations happen at load time. The hot loop touches only what it needs. This is Carmack's philosophy, Muratori's Handmade thesis, and what made id Tech run on hardware it shouldn't have.

Minimalism as correctness

A system is correct not when there is nothing left to add, but when there is nothing left to remove. Every abstraction layer costs memory, cycles, and comprehension. The engine should have zero layers that exist only for elegance. Every interface boundary must earn its existence through performance isolation or genuine platform abstraction.

Deep understanding over convenience

You do not use a collision library. You understand why BVHs work, implement a simple AABB tree, and only reach for a library when you have exhausted your own understanding. The educational goal and the performance goal converge: knowing exactly what your code does lets you optimize it, debug it, and replace it.

Advantages of CPU-first architecture

  • Portability: Software rendering works on any target with a framebuffer — WebAssembly, embedded ARM, old netbooks. No GPU API compatibility nightmares.
  • Determinism: CPU floating point behavior is far more controllable than GPU shader arithmetic. Reproducible simulations and replay systems become tractable.
  • Debuggability: Every pixel value has a clear CPU-side cause. You can single-step through the rasterizer, inspect the z-buffer, watch a triangle being drawn.
  • No API surface area: No OpenGL / Vulkan / Metal / D3D version fragmentation. No driver bugs. No shader compiler variance.
  • Tiny runtime: No GPU context initialization. No shader compilation stall. Engine cold start in milliseconds.
  • Memory ownership: You control every byte. No hidden GPU memory allocations, no VRAM budgets to track separately.

Disadvantages — be honest

Raw throughput ceiling: A modern discrete GPU renders hundreds of millions of triangles per second. A highly optimized CPU software rasterizer on a good 8-core machine might push 5–15 million in real conditions. The gap is 1–2 orders of magnitude. You must design games where this doesn't matter: lower poly counts, orthographic or pseudo-3D views, 2D, or carefully bounded 3D scenes.

  • Resolution limits: Software rendering 1920×1080 at 60 fps with complex shading is genuinely hard. 640×360 or 720×480 upscaled is realistic for potato targets.
  • Modern visual expectations: PBR, screen-space reflections, real-time GI — none are feasible. You will be making a stylistic choice as much as a technical one.
  • Talent perception: Recruiters and some developers will not immediately understand the choice. The portfolio angle requires explanation.

Why modern engines ignore this niche

Because the commercial incentive is photorealism. AAA games sell on visual fidelity. GPU manufacturers fund middleware. Graphics APIs like Vulkan and Metal exist to extract more performance from specialized silicon. The entire industry optimized around "more pixels, more polygons, better lighting" — not "runs on a 2012 laptop." That gap is your market.

Realistic limits in 2026

A modern x86 core can perform ~4–16 billion simple operations per second. With AVX2 SIMD, effective throughput for pixel filling is 8–32× higher. A 720×480 framebuffer is 345,600 pixels — at 60 fps you need 20.7M pixel writes/second, trivially achievable. The real constraint is work per pixel: lighting calculations, texture lookups (cache misses!), depth testing. SIMD rasterization at modest resolutions is well within range of any 4-core CPU from 2015.

Games that benefit most

Simulation-heavy

Dwarf Fortress, RimWorld, Factorio. Most CPU budget goes to simulation, not rendering. CPU-first lets you allocate cores intelligently.

Retro-aesthetic

Doom-style FPS, pixel art RPGs, pseudo-3D racers. The visual constraint is a feature, not a limitation. Growing market on itch.io.

Accessible indie

Games intended for players without gaming hardware. "Runs on anything" is a genuine selling point in low-end hardware markets worldwide.

What "performance by architecture" means in practice

  • Your renderer touches only visible pixels — culling happens before rasterization, not after.
  • Your simulation runs on contiguous arrays of plain data, not graphs of objects with vtables.
  • Your asset pipeline pre-processes everything at build time. The runtime loads and uses — it never converts, parses, or transforms.
  • Your allocator is bump-allocated per-frame scratch, not scattered malloc/free.
  • Your thread model has zero lock contention in the hot path. Synchronization happens at task boundaries, not inside loops.

Is there a real niche?

Yes — but it is specific. The niche is developers who want full control, understand what they're building, target low-end hardware, and value educational depth over rapid prototyping. This is not a mass market. It is, however, a coherent audience:

  • Retro / indie devs: Doom-aesthetic FPS, boomer shooters, pixel art games. Growing market on itch.io.
  • Simulation devs: Games like Dwarf Fortress where CPU budget is better spent on world simulation. Many already use SDL + custom renderer.
  • Educational use: University game programming courses, engine programming self-study. Handmade Hero has ~120K YouTube subscribers. Massive demand for "understand it from scratch."
  • Low-end hardware markets: Southeast Asia, parts of Africa and Latin America have large gaming populations on low-spec hardware.
  • Demoscene / technical art: Developers interested in fitting impressive things into tiny executables.

Why most engines become bloated

  • Feature creep from user requests: Each individual request is reasonable; the aggregate is a monolith. Unity added C# scripting, Bolt, ML-Agents, URP, HDRP, DOTS — each reasonable, collectively incoherent.
  • Abstraction inflation: Every layer added for "flexibility" costs runtime overhead and mental model complexity. Generic enough for everyone = optimized for no one.
  • Historical baggage: Engines must maintain backward compatibility with 10-year-old projects. New architecture cannot replace old without breaking the install base.
  • Talent rotation: Engineers join, add a system their way, leave. No single owner maintains coherent vision. Code entropy accelerates.

Complete Knowledge Graph
Every topic you need — with difficulty, prerequisites, and what "good enough" vs "expert" looks like. Organized into five dependency layers.

Layer 1 — Programming foundations

Advanced C / C++ Foundational

The engine lives here. Use C++ but resist its complexity gravity — prefer C-style code with thin C++ conveniences (constructors, references, templates where beneficial).

Good enough: Pointers, structs, stack vs heap, value semantics, function pointers, basic templates.

Expert: Undefined behavior, strict aliasing, memory model, compiler explorer fluency, ABI stability, placement new, constexpr everything, CRTP.

Common trap: Overusing OOP/inheritance, virtual functions in hot paths, std::shared_ptr everywhere, not reading compiler output.

Memory management Core systems

Every allocation in a game engine is a performance decision. Understanding the heap, stack, virtual memory, and fragmentation lets you build allocators 10–100× faster than malloc.

Good enough: Stack vs heap, malloc internals, fragmentation, basic arena allocator.

Expert: TLSF allocator, slab allocators, buddy system, virtual memory mapping, huge pages, NUMA awareness.

Implement: Arena allocator, pool allocator, frame allocator, header-tracked debug allocator.

CPU architecture Advanced

You are targeting the CPU. Knowing what it does — pipelines, branch prediction, out-of-order execution, cache hierarchy — is not optional. Every optimization decision flows from this.

Good enough: Cache lines (64 bytes), L1/L2/L3 sizes, branch prediction basics, memory latency vs throughput.

Expert: Speculative execution, microarchitecture differences (Zen4, Golden Cove), instruction-level parallelism, store forwarding, false sharing, hardware prefetcher behavior.

Cache locality & data layout Core systems

Cache misses cost 100–300 cycles. A tight inner loop that causes cache misses every iteration will be 10–30× slower than one that doesn't. This single topic explains most performance gaps between naive and expert engine code.

Good enough: SoA vs AoS, avoid pointer-chasing in hot loops, know your data's size.

Expert: Cache-oblivious algorithms, Morton order, hot/cold data splitting, prefetch instructions, measuring with perf / VTune.

Layer 2 — Systems programming

Data-oriented design (DOD) Core systems

DOD inverts OOP thinking: instead of "objects that have data," think "streams of data that are processed." ECS is DOD's most visible application in games.

Good enough: Flat arrays over linked lists, separating hot data from cold, thinking in terms of transforms on arrays.

Expert: Archetype ECS with chunk iteration, automatic SoA layout, query scheduling, system dependency graphs.

SIMD (SSE / AVX / NEON) Advanced

The single highest-leverage optimization in a software renderer. Processing 8 pixels simultaneously with AVX2 is an 8× throughput multiplier on the rasterizer's inner loop. For a CPU-first engine, SIMD fluency is non-negotiable.

Good enough: 4-float SSE operations, alignment requirements, intrinsic functions.

Expert: AVX-512 gather/scatter, horizontal reductions, SIMD-friendly data structures, NEON equivalents for ARM.

Multithreading & atomics Advanced

Modern CPUs have 8–32 cores. A single-threaded engine wastes 87–97% of available compute.

Good enough: Mutex/semaphore, thread pool, basic producer-consumer queue, memory ordering (acquire/release).

Expert: Lock-free queues (SPSC, MPMC), fiber-based job systems, work-stealing schedulers, hazard pointers, wait-free data structures.

Common trap: Adding locks everywhere "to be safe." Every lock is a potential contention point. Design for lock-free where possible; synchronize at task boundaries, not inside loops.

Operating system internals Core systems

Platform abstraction layers require knowing what each OS actually provides: file I/O, virtual memory, threading, process model.

Good enough: Virtual memory, file descriptors, POSIX basics, Windows API basics, process/thread model.

Expert: mmap internals, page fault handling, OS scheduler behavior, system call overhead, huge TLB pages, memory-mapped files for asset loading.

Layer 3 — Rendering & math

Math for graphics Foundational

Vec2/3/4, Mat3/4, quaternions, euler angles, perspective/orthographic projection, view frustum, dot/cross product, barycentric coordinates, ray-triangle intersection, spatial transformations.

Implement your own vec3, mat4, and quaternion math first. Then you may use a library. You cannot optimize what you don't understand.

Software rasterization Expert

This IS your renderer. Scanline rasterization, edge equations, barycentric interpolation, subpixel precision, fill rules, texture mapping at the CPU level.

Good enough: Draw a triangle to a framebuffer, perspective-correct texture mapping, depth buffer.

Expert: SIMD tile rasterizer, hierarchical z-buffer, early-out optimizations, sub-pixel accuracy, coverage masks.

BSP trees & spatial structures Advanced

Doom and Quake used BSP trees to solve visibility without a z-buffer. Understanding spatial partitioning (BSP, BVH, octree, kd-tree) is fundamental to both rendering and physics.

Good enough: BSP concept, AABB tree, BVH construction.

Expert: SAH BVH, portals + PVS, SVDAG for sparse voxels, octree streaming.

Scripting systems Advanced

Game logic should not require recompiling the engine. A scripting layer — embedded Lua, a custom bytecode VM, or a hot-reloaded C plugin system — separates engine from game code.

Good enough: Embed Lua or a small language, expose ECS through a C API.

Expert: Write your own bytecode VM, implement a simple compiler, design a reflection system that auto-generates bindings.

Layer 4 — Engine systems

ECS (Entity Component System) Core systems

ECS is the dominant architecture for performance-critical game simulations. Systems iterate over contiguous component arrays, enabling SIMD and cache-friendly processing.

Good enough: Sparse-set ECS (EnTT-style), basic query iteration.

Expert: Archetype ECS (Unity DOTS-style), chunk-based iteration, parallel system scheduling.

Study the EnTT source code. It's the gold standard of sparse-set ECS in C++ and the codebase is genuinely educational.

Asset pipeline & serialization Core systems

Raw assets (PNG, OBJ, WAV) are not engine assets. Your pipeline converts them at build time to memory-layout-optimal binary — directly loadable with minimal parsing.

Good enough: PNG→raw RGBA, OBJ→your mesh format, WAV→raw PCM, simple custom binary format.

Expert: Incremental build system, content-addressed storage, streaming/partial asset loading, hot reload of assets.

Audio system Core systems

Audio is a real-time system on a separate thread with strict latency requirements. Understanding the audio callback model, PCM mixing, and basic DSP is essential.

Good enough: Mix 32 voices of PCM audio, basic spatial panning, volume envelope.

Expert: FFT convolution reverb, HRTF spatialization, interactive music systems, custom DSP graph.

Hot reload & reflection Advanced

Recompiling and relaunching to test a gameplay change is a development speed killer. Hot reload dramatically improves iteration. Reflection enables editors, serialization, and scripting bindings.

Good enough: Hot-reload assets. Game code as a shared library, reloaded on change.

Expert: Stateful hot-reload preserving entity state, code patching, compile-time reflection with code generation.

Layer 5 — Cross-platform & tooling

Build systems Core systems

CMake as baseline. Understand compilation units, link stages, static vs dynamic libraries. Learn to minimize rebuild time through unity builds and forward declarations.

WebAssembly Advanced

Emscripten toolchain, memory model differences, JavaScript interop, canvas framebuffer access, threading with SharedArrayBuffer, audio worklet for low-latency audio.

Platform abstraction Core systems

Thin OS layer exposing: window creation, framebuffer blit, input events, file I/O, thread primitives, timer. Everything above this layer is platform-agnostic.

Profiling & debugging Core systems

CPU profilers (perf, VTune, Tracy), memory profilers (Valgrind, ASan), custom in-engine overlay. You cannot optimize what you cannot measure. Build profiling in from day one.

Compiler internals Advanced

Understanding what the compiler does to your code — assembly output, optimization reports, PGO, LTO, sanitizers — lets you write code that compiles well and runs fast. godbolt.org is essential.

Custom allocators Core systems

Linear/arena, stack, pool, free-list, buddy system. Profile before choosing. An arena with a single pointer increment is fastest for per-frame temporaries. A pool is fastest for fixed-size components.

Platform layer

Design: thin OS wrapper, never leaked upward

The platform layer is the only code that differs per-platform. It exposes exactly this interface — and nothing more:

  • platform_create_window(w, h, title)
  • platform_blit_framebuffer(fb)
  • platform_get_input_events()
  • platform_get_time_ns()
  • platform_file_read(path, dest_arena)
  • platform_thread_create / mutex / semaphore / atomic

Everything above the platform layer is testable without a running OS. The platform layer is the seam for WASM, macOS, Linux, and Windows implementations. If it requires understanding the game, it's not platform code.

Memory architecture

Permanent arena

Allocated at startup, never freed during runtime. Contains: entity component storage, loaded asset data, ECS archetype buffers. Single large OS allocation upfront (e.g. 512 MB). Bump pointer only moves forward.

Frame scratch

Reset every frame. Used for: render command lists, temporary sort buffers, string building, any per-frame allocation. Zero-cost "free" — just reset the pointer. 64 MB is generous. Never use for anything surviving a frame boundary.

Asset pool

Fixed-size slots for assets (textures, meshes, sounds). Pool allocator: O(1) alloc/free, zero fragmentation. Max N of each type defined at startup based on the game's asset count. No dynamic resizing at runtime.

Debug/editor heap

Non-shipping builds only. Uses system malloc for developer tooling. Never called from the game loop hot path. If you see a malloc in a profiler trace, something is architecturally wrong.

Renderer architecture

CPU software renderer — design principles

Framebuffer ownership: The renderer owns two framebuffers (double-buffered). Game writes to back buffer; platform layer blits front buffer to screen. Swap at vsync or at fixed intervals.

Tile-based rendering: Divide the screen into 64×64 or 32×32 tiles. Assign tiles to worker threads. Each tile is independent — no inter-tile communication in the rasterizer hot path. Near-linear scaling with core count.

Render command list: The game produces a list of draw commands (draw_mesh, draw_sprite, draw_text) into frame scratch memory. The renderer sorts and executes this list. Decouples game simulation from rendering, allows culling before rasterizing, and enables draw order optimization.

Texture cache strategy: Sequential screen pixels jump across texture memory in standard row-major layout. Morton-order (Z-order curve) texture storage significantly improves locality. Pre-generate mipmaps in your asset pipeline.

ECS design

Recommendation: sparse-set ECS to start, archetype ECS later

Sparse-set ECS (EnTT-style): Entities are integer IDs. Each component type has a sparse array (ID → component index) and a dense array of components. Iteration over a component type is always linear. Insertion and removal are O(1). Simpler to implement.

Archetype ECS (Unity DOTS-style): Entities with identical component sets are grouped into archetypes. Components stored in SoA chunks. Much better cache behavior for multi-component queries. More complex — add in Phase 4.

What not to do: Virtual dispatch in component update loops. A vtable call per entity kills cache line coherency and branch prediction. Systems iterate over arrays, not objects.

What to intentionally NOT overengineer

Scene graph: A flat entity array with parent/child IDs and a simple sorted transform propagation loop is sufficient for 95% of games. Complex scene graphs with deep inheritance are a Unity-era mistake that costs performance and comprehension.

  • Plugin system: Don't build one until you have 3 real use cases. Interfaces designed in advance are wrong interfaces.
  • Networking: Explicitly out of scope. Do not add it.
  • Physics engine: Start with AABB collision. Full rigid body only if a game requires it.
  • Shader system: You don't have shaders. CPU rendering means "shading" is C functions. Don't add a shader abstraction layer.
Section 6
Software Rendering Masterpath
The deep dive into CPU rendering — what's still relevant, what changed, and how to build an excellent software renderer.

The rendering pipeline — CPU edition

  1. Frustum culling: Reject objects entirely outside the view frustum. AABB vs plane tests. Trivially parallelizable and eliminates most scene geometry before rasterization.
  2. Occlusion culling: Reject objects hidden behind others. Software HiZ buffer approach: use the previous frame's depth buffer (downsampled) to test object AABB depth. If the object's nearest depth is behind the HiZ value, skip it.
  3. Vertex transform: Model → World → View → Clip space for each vertex. SIMD-accelerate 4 vertices at a time with AVX.
  4. Clipping: Clip triangles against the near plane. Sutherland-Hodgman algorithm generates new vertices at clip boundaries.
  5. Rasterization: Determine which pixels each triangle covers. Compute barycentric coordinates. Interpolate attributes (UV, normal, color). Run the fragment function (texture lookup + lighting). Write color + depth.
  6. Post-processing: Optional fullscreen passes: dithering, color grading, scanline overlay, CRT simulation — all cheap on CPU at low resolutions.

Triangle rasterization — the right algorithm

Scanline rasterizer (old approach)

For each row between triangle top and bottom, compute left and right x bounds and fill the span. Simple to implement. Hard to SIMD-accelerate efficiently, poor cache behavior on wide triangles.

Doom/old style
Edge equation rasterizer (preferred)

Three edge functions define the triangle interior. For each pixel in the bounding box, evaluate all three simultaneously. Embarrassingly parallel — evaluate a block of 8×8 pixels with SIMD. This is what modern GPU rasterizers do.

Recommended for SIMD

Tile-based approach: Divide screen into 64×64 tiles. Bin each triangle into overlapping tiles (prepass). Dispatch tiles to worker threads. Each thread rasterizes its tile's triangles independently — no locking, linear cache access. This is how console GPU rasterizers work and maps perfectly to your job system.

Texture mapping

Perspective-correct texture mapping — why it matters

Linear interpolation of UV coordinates in screen space is mathematically wrong — it produces the classic "swimming texture" artifact from early 3D games. Perspective-correct interpolation requires dividing UV by W at each vertex, interpolating the divided values, then multiplying back by interpolated 1/W per pixel. ~2 extra operations per pixel — worth it.

Texture cache optimization:

  • Morton-order (Z-curve) textures: Nearby screen pixels map to nearby texture memory. 2–4× speedup on large textured polygons.
  • Mipmaps: Pre-computed lower-resolution versions. Select mip level based on screen-space derivatives. Prevents aliasing and reduces bandwidth at distance. Generate in the asset pipeline.

Doom-style renderer — what's still relevant

BSP tree + sector/portal system

Doom split the level into convex sectors using a BSP tree. Rendering traversed front-to-back from the camera, drawing each sector's walls once with no overdraw. For levels with many rooms and corridors, this is still extremely efficient. PVS (Potentially Visible Set) — precomputed visibility between sectors — extends this further.

Still highly relevantInterior environments
Column-based rendering

Wolfenstein 3D rendered the world as columns of pixels — each vertical strip corresponds to a ray cast into the scene. Each column is independent and trivially parallelizable. Works only for flat-floor, vertical-wall worlds but is extremely fast. PICO-8 raycasters use this approach.

Still relevantPseudo-3D games

Lighting on CPU

  • Lightmaps (precomputed): Bake lighting offline into a texture. Runtime cost: a single texture sample per pixel. This is how Quake lit levels — beautiful results, zero runtime cost. Your asset pipeline bakes lightmaps using a path tracer.
  • Vertex lighting: Compute lighting per-vertex, interpolate across triangle. Very fast, low fidelity, works well for low-poly art styles.
  • Flat + dynamic lights: Flat-shaded geometry with 4–8 dynamic point lights. Precompute which triangles each light affects. At runtime, only update affected triangles.
  • Palette tricks: Doom used a single "colormap" — a 256-color lookup table mapping palette indices to darker ones as distance increased. No math — just a table lookup. Extremely fast, iconic aesthetic.

SIMD acceleration of the rasterizer

AVX2 8-wide triangle rasterizer — the core optimization

The edge equation test for 8 pixels in a tile simultaneously:

  • Load 8 pixel x-coordinates into a __m256i register
  • Evaluate all three edge functions for all 8 pixels simultaneously
  • Create a coverage mask with SIMD comparison
  • For covered pixels: interpolate UVs, fetch texels, compute lighting
  • Write 8 pixels to the framebuffer with a masked store

A 1-wide scalar rasterizer might rasterize 30–50M pixels/second. An 8-wide AVX2 rasterizer can reach 200–400M pixels/second on the same hardware. For a 640×480 framebuffer at 60 fps, you need 18.4M pixel writes/second — SIMD headroom lets you do far more work per pixel.

What became obsolete

  • Sorting-based BSP rendering (no depth buffer): Only necessary when depth buffer memory was expensive. Today, 4 MB of depth buffer is trivial.
  • Palette-based color (8-bit indexed): Obsolete for compatibility, but still aesthetically valid and actually faster than 32-bit for fill operations.
  • Fixed-point math: Necessary on CPUs without FPUs. Modern CPUs have fast floating-point units. Use floats — unless targeting actual embedded hardware without an FPU.

Hybrid CPU/GPU rendering

Optional acceleration path: On machines with integrated graphics, use the GPU to blit the framebuffer with GPU-accelerated upscaling (e.g. from 640×360 to 1920×1080). Zero rendering logic on GPU — just a texture upload and a fullscreen quad draw. The CPU does all rendering; the GPU handles final display scaling. ~50 lines of code using SDL2's texture API.

Section 7
Performance Engineering
The deep principles of making code fast — not through tricks, but through understanding what the hardware actually does.

The performance hierarchy

In order of impact, highest to lowest:

  1. Do less work — Culling before rasterizing is 100× better than a faster rasterizer. Precomputing offline is infinitely better than computing at runtime. Algorithm choice beats micro-optimization every time.
  2. Data layout — Cache-friendly data structures (SoA, flat arrays) beat smart algorithms on cache-unfriendly data. A linear scan of an array beats a binary search in a linked list for small N.
  3. Parallelism — Using 8 cores instead of 1 is an 8× multiplier. Job systems, tile rendering, and parallel asset loading all belong here.
  4. SIMD — Processing 8 values simultaneously is an 8× multiplier on the specific operation. Apply where data parallelism exists and the operation is already a confirmed bottleneck.
  5. Micro-optimization — Branch prediction, instruction scheduling, instruction fusion. Meaningful only after everything above is done.

Cache locality — the most important concept

Memory latency reality

L1 cache hit: ~4 cycles. L2 hit: ~12 cycles. L3 hit: ~40 cycles. RAM miss: ~200 cycles. One cache miss to RAM costs as much as 50 L1 hits. An inner loop that misses cache every iteration is 50× slower than it needs to be.

A cache line is 64 bytes. When you access any byte, 64 bytes load. If your struct is larger than 64 bytes but you only use 8 bytes, you're wasting 87% of cache bandwidth.

Hot/cold data splitting

If an entity has Position (12 B), Velocity (12 B), Health (4 B), AI State (200 B) — and your movement system only touches Position + Velocity — don't store them together. With mixed layout: 0.5 entities per cache line. With split layout: 5 entities per cache line. 10× cache efficiency from one struct split.

SoA vs AoS

AoS (Array of Structures) — typical OOP layout:

struct Entity { float x, y, z; float vx, vy, vz; int health; }; Entity entities[N];

Sequential entities are adjacent. Good if you always process all fields of one entity before moving to the next. Bad for SIMD (x values of different entities are 32 bytes apart).

SoA (Structure of Arrays) — DOD layout:

float positions_x[N], positions_y[N], positions_z[N];

All x positions are adjacent. SIMD loads 8 x-values in one AVX register. A position-only system has perfect cache behavior. Bad if you always need all fields of one entity.

Rule: Default to AoS. Switch to SoA only when a system touches a subset of fields for many entities AND profiling confirms cache misses are the bottleneck.

Branch prediction

Modern CPUs predict branch outcomes and speculatively execute the predicted path. A correctly predicted branch costs 0 cycles. A mispredicted branch costs ~15 cycles (pipeline flush).

  • Sort data before processing to make branches predictable (all alive entities first, then dead).
  • Replace unpredictable branches with branchless computation where possible (conditional moves).
  • For SIMD, use masks instead of branches — compute both paths, blend results with a mask.
  • Don't add [[likely]] / [[unlikely]] hints without profile data to justify them.

False sharing

The silent multithreading killer: If two threads write to different variables that share a cache line, the CPU must coordinate cache ownership between cores — even though they're writing to different addresses. This can cause 10–50× slowdowns on code that looks correctly parallel. Fix: pad shared data structures to cache line boundaries. alignas(64) struct WorkerState { ... };

Custom allocators — the actual performance numbers

malloc: ~50–200 cycles per allocation (fragmentation, lock contention, bookkeeping). At 10,000 allocations per frame at 60 fps: 30–120M cycles/sec spent just in the allocator. On a 4 GHz CPU, that's 7–30ms per second — meaningful overhead.

Arena allocator: 1–3 cycles per allocation (single pointer add + bounds check). 0 cycles to free (pointer reset at end of frame). The difference is real and measurable.

  • Per-frame temporaries → Frame arena (reset each frame)
  • Fixed-lifetime assets → Pool allocator
  • Long-lived game state → Permanent arena
  • Miscellaneous/unpredictable → Never in the hot path. If you need malloc in a hot loop, you have an architectural problem.

What indie engines usually get wrong

Premature abstraction: Adding an "event system" or "message bus" before understanding access patterns. Indirection + allocation per event destroys cache performance. For high-frequency events (per-entity per-frame), use direct function calls or flags in component arrays. For low-frequency events (button click, scene load), a simple callback list is fine.

  • Using std::vector of pointers: All your data is scattered in memory. Allocate into contiguous arrays and use indices.
  • Profiling only frame time: Frame time tells you something is slow, not where. Use a sampling profiler that shows the call graph and cache miss rates.
  • Optimizing the wrong thing: A week optimizing a 0.5ms system when your bottleneck is a 10ms texture load. Measure first, always.
Section 8
Recommended Technology Stack
What to use, what to replace later, and precisely why each recommendation exists.

Language

C++ (C-style subset) — primary language

Why C++: SIMD intrinsics, manual memory control, zero-overhead abstractions, compile-time computation, direct OS API access, no GC pauses, massive tooling ecosystem. The performance control needed for CPU-first rendering requires C or C++.

Which features to use: structs/classes (no virtual), templates (sparingly), references, constexpr, [[nodiscard]], designated initializers (C++20). Avoid: exceptions, RTTI, complex inheritance, std::shared_ptr, heavy STL in hot paths.

Scripting: LuaJIT preferred. LuaJIT's performance is exceptional. Lua's C embedding API is well-documented and minimal. Alternative: custom bytecode VM after Phase 4.

Build system

CMake — start here
Phases 0–3

Industry standard, cross-platform, supported by every IDE and CI system. Use modern CMake (target_* style), avoid legacy global flags. Add compile_commands.json generation for clangd IDE support.

Custom build script — later
Phase 5+

A single-file build script in Python or C achieves faster incremental builds by understanding your specific source tree. For this engine's scope, well-structured CMake is sufficient indefinitely.

Libraries — start with, then replace

LibraryPurposeReplace withWhen
SDL2 / SDL3Window, framebuffer blit, input, audio initCustom platform layer per-OSPhase 7
miniaudioAudio device + callbackCustom per-platform audio (WASAPI, CoreAudio, ALSA)Phase 5 if needed
stb_imagePNG/JPG loading in asset pipelineNever at runtime — pipeline tool onlyNever

Compilers

Clang Primary

Best diagnostics, best sanitizers (ASan, UBSan, TSan), best compile times for large codebases, LLVM backend for WASM. Use for development.

GCC Linux

Often generates better AVX code than Clang for tight loops. Profile both and pick the winner per platform. Required for Linux distribution.

MSVC Windows

Required for Windows compatibility testing. Consider ClangCL (Clang targeting MSVC ABI) as a middle ground with better diagnostics.

Profilers

Tracy Profiler Highly recommended

Open-source, low-overhead, real-time frame profiler. Instrumentate code with simple macros. Connects over network to a separate profiler UI. Shows frame timing, zone timings, memory allocations. The best free profiler available.

Platform profilers

Linux: perf + hotspot UI, VTune (Intel, free for open source). Windows: Windows Performance Analyzer, Superluminal. Mac: Instruments (Xcode). Use these to find cache miss patterns Tracy doesn't show.

Debuggers

  • GDB / LLDB: Command-line debuggers. Learn them well — you'll use them constantly. TUI mode makes GDB bearable.
  • RemedyBG (Windows): Excellent native debugger, much faster than Visual Studio's for large codebases.
  • rr (Linux): Record-and-replay debugger. Record a crash, replay it deterministically, step backward. Invaluable for heisenbugs.
  • AddressSanitizer / UBSanitizer: Compile with -fsanitize=address,undefined. Catch memory errors and undefined behavior before they become mysterious crashes.

Cross-platform & CI

  • Emscripten: LLVM-based C++ → WASM compiler. The only viable path for browser deployment. SDL2 has an Emscripten backend; most system calls are emulated.
  • GitHub Actions: Free CI for open-source. Build matrix: Windows / Linux / macOS + Clang / GCC / MSVC. WASM build in CI catches portability regressions early.
  • Zig as a cross-compilation toolchain: zig cc wraps Clang with a complete hermetic toolchain. Build a Linux ARM binary from Windows. Surprisingly practical.
Section 9
Practical Build Order
The exact sequence to build the engine — with milestones, traps, and scope limits at each stage.
  • 1
    First executable — hello framebuffer

    Open a window using SDL2. Fill a pixel buffer with a color. Blit it to the window. Measure frame time. Exit cleanly.

    Success: A colored window at stable 60 fps. You understand the game loop, fixed timestep, and platform blit.

    Scope limit: No input, no renderer. Just the loop and the blit. Complexity enters one system at a time.

  • 2
    First renderer — 2D software drawing

    Draw filled rectangles, circles, and lines to the framebuffer. Load a PNG with stb_image, draw it as a sprite. Add a z-layer concept. Add alpha blending.

    Common trap: Over-abstracting the renderer interface before you know what the renderer needs. Write concrete functions first: draw_rect(), draw_sprite(). Abstract later when patterns emerge.

  • 3
    First game — Pong or Breakout

    Complete playable game: input handling, game state machine, collision, scoring, win/lose condition. Everything in C, software renderer only.

    Success: A real person plays and understands the game. Code is under 2,000 lines.

    Refactor signal: If adding a feature requires changing 5+ files, your architecture has too much coupling. Flatten it.

  • 4
    First 3D renderer — spinning textured cube

    Implement a 3D perspective camera, vertex transform pipeline, triangle rasterizer with depth buffer, perspective-correct texture mapping.

    Success: A textured cube rotating smoothly. No z-fighting. 30+ FPS at 640×480 on a 2015-era machine.

    Common trap: Debugging with complex scenes. Start with one triangle, then one quad, then the cube. Add complexity incrementally.

  • 5
    First ECS — entities with components

    Implement a sparse-set ECS. Create entities, attach components (Transform, Velocity, Sprite), iterate in systems. Re-implement your Pong on the ECS.

    Success: Identical gameplay, cleaner architecture, measurably faster iteration over entities.

  • 6
    First asset system — binary asset format

    Write an offline asset converter: PNG→.tex (raw RGBA binary), OBJ→.mesh (packed vertex binary). Runtime loads binary directly with minimal parsing.

    Success: Asset load time drops 5–20× vs parsing raw PNG/OBJ at runtime. Load time measured and documented.

  • 7
    First Doom-style level — raycasting or BSP

    Build a level format (2D grid or BSP map). Implement raycasting (Wolfenstein-style) or BSP traversal (Doom-style). Walk through the level with a first-person camera. Texture-mapped walls, floor, ceiling.

    Success: You can walk through a level that feels like Wolfenstein 3D or Doom. Stable frame rate at target resolution.

  • 8
    First audio — PCM mixing

    Integrate miniaudio. Implement a mixer: N simultaneous mono/stereo sounds at configurable volume and pitch. Simple volume falloff with distance.

    Success: Sound effects and music play correctly with no crackling. Correct looping. Latency <50ms.

  • 9
    First optimization pass — profiler-guided

    Integrate Tracy. Profile a full game session. Identify the top 3 bottlenecks. Optimize each with concrete changes (SoA layout, culling, SIMD pilot). Measure improvement.

    Success: Documented: initial bottleneck (where, why), change made, before/after numbers. This documentation IS the portfolio piece.

    Refactor vs continue: Refactor when a design actively prevents the next milestone. Continue when the design is "not ideal but works." Refactoring for aesthetics costs weeks. Refactoring to unblock capability is essential.

  • 10
    First scripting layer — Lua game logic

    Embed LuaJIT. Expose entity creation, component access, input, and audio through the Lua C API. Rewrite enemy AI and game rules in Lua. Hot-reload scripts while game runs.

    Success: Changing enemy behavior requires editing a .lua file and saving. Game updates within 1 second without restart.

  • 11
    First editor — minimal tooling

    Simple in-engine overlay: entity list, component inspector, asset browser, console. Level editor: place entities, configure properties, save/load level.

    Success: You can design a level without touching code. A hypothetical non-programmer could place and configure entities.

  • 12
    First cross-platform port — WASM

    Compile to WebAssembly with Emscripten. Game runs in a browser at target frame rate. Share a URL, game loads in <5 seconds.

    Success: Anyone with a browser can play your game without installing anything. Your most powerful distribution story.

  • 13
    First game demo — complete experience

    3–5 minute playable demo. Defined start and end. Working audio. Polished enough to show publicly. Submitted to itch.io and shown in a portfolio.

    Success: External players complete the demo without guidance. Frame rate stable on a 2015-era machine. Executable under 10 MB.

Section 10
Long-Term Specialization Paths
After the core engine is complete, where can you take it? Each path is a multi-month deep dive.
Retro FPS engine
6–12 months

Push the Doom-style renderer to its limits. Implement Quake-style BSP with PVS. Add decals, dynamic lights, particle effects. Build a level editor in the spirit of TrenchBroom. This path leads to a publishable "game creation kit" for boomer shooter games — a real market on itch.io and Steam.

Deterministic simulation engine
6–12 months

Replace floating-point math with fixed-point. Implement a deterministic physics engine. Design a lockstep replay system — record all inputs, replay them, get bit-identical output. Enables complete game replays from inputs only, deterministic multiplayer, debugging tools that can replay any bug. Highly valued for RTS, fighting game, and simulation genres.

Tiny executable engine
3–6 months

LTO + strip + upx compression + custom startup code. 4KB intros, 64KB demos, 256KB games. Learn: custom PE/ELF headers, syscall-based platform layer without CRT, Crinkler/kkrunchy compressors, procedural audio, procedural graphics. Portfolio showpiece of extreme technical skill.

CPU raytracer / path tracer
12+ months

Replace the rasterizer with a path tracer for offline rendering — cinematic screenshots, asset pipeline previews, or a "cinematic mode." Start with Peter Shirley's "Ray Tracing in One Weekend." Then: SIMD BVH traversal, wavefront path tracing, importance sampling, ReSTIR. A career specialization unto itself.

Custom scripting language + compiler
12–24 months

Design and implement your own scripting language. Lexer → parser → AST → bytecode compiler → VM interpreter. Requires formal language theory, compiler design (Crafting Interpreters is the best resource), and bytecode VM design. A typed Lua with structs and optional GC is achievable in 12 months. The deepest technical rabbit hole on this list.

ECS research & archetype engine
6–9 months

Deep dive into ECS architecture research. Implement archetype ECS from scratch. Study Flecs source code. Design a system query language. Implement parallel system scheduling with dependency graphs. Compare archetype vs sparse-set performance across real workloads. Write a technical blog post series. This is publishable research-quality work.

Embedded / low-power game systems
6–12 months

Port the engine to ARM microcontrollers (Raspberry Pi Pico, STM32 with display) or Raspberry Pi. Handle: no FPU (fixed point), no OS (bare metal), tight RAM (256KB–2MB), slow display I/O (SPI bus to TFT screen). The ultimate constraint-driven design — every byte matters, every cycle counts.

Demoscene optimization
Ongoing

Study and participate in demoscene competitions. Categories: 4KB intro, 64KB intro, 256KB demo. Techniques: procedural generation, timeline editors, executable compression, procedural audio. The demoscene is the world's most extreme applied computer science community — the people who truly push hardware to its limits.

Section 11
Reality Check
Brutally honest assessment of what this project actually requires — and what will kill it.

Time investment — honest estimates

Building a complete, polished, cross-platform engine takes 3–7 years of serious part-time work (10–20 hours/week). This is not pessimism — it's the honest history of every solo engine project that succeeded.

  • Phase 0–1 (Foundations): 200–400 hours to genuinely understand, not just complete.
  • Phase 2 (Software rendering): 300–500 hours for a real 3D renderer.
  • Phase 3–4 (Engine core + optimization): 400–800 hours.
  • Phase 5–8 (Advanced through expert): 800–2,000+ hours.

Total: 1,700–3,700 hours to a genuinely expert-level, complete engine. At 15 hours/week, that's 2.2–4.8 years. This is a lifestyle project, not a semester project.

What kills solo engine projects

The feature spiral

You add scripting before the renderer is stable. The scripting system exposes renderer bugs. You fix the renderer, which breaks the scripting API. You redesign the scripting API. Progress stalls and frustration mounts.

Fix: finish phases completely before starting the next.

Premature perfection

Spending 3 months designing the "perfect" ECS architecture before implementing anything. The architecture will be wrong. You only learn what you need from building.

Fix: build the worst version that works, then replace it with something better informed by reality.

Scope creep

Adding online multiplayer "because it would be cool." Adding procedural terrain before finishing the basic renderer. Every feature outside the core vision delays everything inside it.

Fix: write a scope document. Say no to everything not in it.

Comparing to production engines

Comparing your 2,000-line platform layer to Godot's million-line codebase and feeling inadequate. Godot has 30+ years of combined developer time.

Fix: compare to your past self, not to a team's output. Measure growth, not absolute size.

What actually matters most

Getting a game running on your engine matters more than the engine's architecture. An engine that has shipped one small game, however ugly, is infinitely more valuable than a beautiful architecture diagram for an engine with no game. The game is the validation.

What can be skipped initially

  • Physics engine: AABB collision is enough for Phases 0–4. Full rigid body only if your game requires it.
  • Animation system: Simple frame-based sprite animation is fine for most game types. Skeletal animation is a significant implementation investment.
  • Networking: Out of scope. Do not add it.
  • Visual scripting: Text scripting first. Visual scripting is an editor feature, not an engine feature. Phase 6+ at earliest.

What to learn deeply — do not skip

  • Memory management: Understanding allocation, fragmentation, and custom allocators is non-optional. Every other optimization depends on this.
  • CPU architecture: Cache behavior, branch prediction, and SIMD are not optional extras — they are the core skill of this project.
  • Profiling: If you haven't measured it, you don't know it's slow. Never optimize without a profile.
  • Rasterization math: Perspective-correct interpolation, fill rules, subpixel precision. Getting this right is the renderer. Getting it wrong produces subtle bugs that haunt you forever.

Which optimizations are actually meaningful

High impact — do these
  • Frustum culling before rasterization — eliminates most triangles
  • Tile-based rendering — enables parallelism across cores
  • Binary asset format — eliminates load-time parsing cost
  • Arena allocators — eliminates malloc overhead in hot paths
  • SIMD rasterizer inner loop — 4–8× throughput on pixel processing
Only with profile data
  • Manual loop unrolling (compiler usually handles this)
  • Inline assembly (rarely better than intrinsics + compiler)
  • Branch hints without measurements
  • Micro-optimizing anything not in a measured hot path
Section 12
Final Recommended Path
The optimal approach for this specific engine — distilled to actionable decisions.

Optimal first steps

  1. Read "Computer Systems: A Programmer's Perspective" chapters 1–6. This is the foundation for everything else. The memory hierarchy chapter alone will change how you write code.
  2. Watch Handmade Hero episodes 1–20. Casey builds a platform layer and game loop from scratch. This directly maps to your Phase 0–1.
  3. Write a BMP file writer in C with zero libraries. ~100 lines. Forces you to understand file I/O, pixel formats, and the gap between "knowing C" and "writing systems code."
  4. SDL2 window + framebuffer blit + game loop. Your first running "engine." Measure frame time. Understand vsync.
  5. Pong. Build it completely. This is your Phase 1 graduation.

Ideal first playable demo

A Wolfenstein 3D / Doom-style corridor shooter — 3–5 minutes of gameplay

Why this specific game type:

  • Validates the entire software renderer (raycasting or BSP)
  • Requires ECS — player, enemies, pickups, doors
  • Requires audio — footsteps, gunshots, ambient
  • Requires a level format and basic editor
  • Runs at 320×240 upscaled — trivially within CPU rendering budget
  • Has a clear aesthetic that audiences immediately recognize and respect
  • Can be compiled to WASM for instant browser demos

This demo, shared on itch.io with a technical blog post explaining the software renderer, is an extremely strong portfolio piece. It demonstrates: systems programming, graphics math, cross-platform engineering, and performance optimization — all in one artifact.

Ideal first commercial-quality prototype

A simulation / strategy game — a tiny RimWorld or Dwarf Fortress

Why this type validates a CPU-first engine better than any other:

  • Simulation budget (pathfinding, AI, resource tracking) uses the CPU budget that would otherwise go to rendering
  • Visual style (pixel art, orthographic, or pseudo-3D) aligns with CPU rendering's natural aesthetic
  • Save/load, hot reload, and scripting systems all get tested
  • Demonstrates the engine is not just a renderer demo but a complete game development platform

Best long-term architecture choices

Keep these
  • Platform layer as the only platform-specific code
  • Arena allocators for all runtime allocations
  • Sparse-set ECS (upgrade to archetype when needed)
  • Binary asset format with offline build pipeline
  • Lua scripting over custom VM until Phase 8
  • Tile-based software renderer from the start
Avoid these
  • Complex inheritance hierarchies anywhere
  • Virtual functions in hot paths
  • Reference counting / smart pointers in game code
  • malloc / free in the game loop
  • Deeply nested callbacks for high-frequency events
  • Any feature without a concrete game use case

What should remain intentionally minimal

The editor. Build the minimum editor that enables game creation. A scene hierarchy + property inspector + asset browser is 90% of what game creation needs. The remaining 10% (visual scripting, animation editor, terrain tools) should be added only when a specific game requires it. Editor complexity is a black hole for solo engine development time.

  • The asset format: Simple binary header + raw data. Add versioning. Don't add complex schema evolution until you have multiple shipped games that need it.
  • The scripting API: Expose only what game code needs. Every API endpoint is a maintenance obligation. Start small, expand from real game requirements.
  • The physics system: AABB + swept collision for 90% of games. Add rigid body only when a game explicitly needs it.
  • The audio system: PCM mixing of 32 voices. Add DSP effects only for games that use them. Do not build a general-purpose DSP graph for an engine where no game has yet needed one.

The most important thing

Ship a game. Not a demo. Not a renderer tech demo. A game, however small, with a start and an end, that someone else can play without your help. This single constraint will drive more engine design decisions than any architecture document ever written. The engine serves the game. If a system doesn't help you ship the game, it's not part of the engine yet. Every great engine grew from a game that needed it — not from a spec that predicted what a game would need.
