Phase 0 — Foundations
Months 1–3 · C, memory, and systems thinking. No libraries. No shortcuts.
5 mini-projects
1
BMP file writer from scratch
Easy
Write a 24-bit BMP image file using only C file I/O — no libraries at all. Open a file, write the 54-byte BMP header manually by filling a struct with exact field values, then write a 256×256 grid of RGB pixels. Try rendering a gradient or a checkerboard pattern. The goal is to experience the gap between "knowing C" and "writing binary data correctly."

Dynamic array (growable buffer)
Easy
Implement a dynarray in C that wraps a heap-allocated buffer: push appends an element and doubles capacity when full, get retrieves by index, free releases memory. Then implement the same thing but generic using void* and element size. Write tests that push 10,000 integers and verify all values are correct after retrieval.
Heap allocationRealloc strategyGeneric C with void*Capacity doubling
3
Hash map with open addressing
Medium
Build a hash map that maps string keys to integer values. Use an array of buckets, a djb2 or FNV-1a hash function, and linear probing for collision resolution. Implement insert, lookup, and delete. Handle tombstone entries for deletion. Benchmark it against naive linear search for 10,000 lookups and confirm hash map wins at that count.
Hash functionsOpen addressingLoad factorTombstones

Arena allocator
Easy
Write an arena allocator that takes a large upfront buffer (e.g. 64 MB from malloc), maintains a single bump pointer, and hands out aligned chunks on request. Add a reset() that zeroes the pointer back to the start. Then benchmark it: allocate and reset 1,000,000 times and compare total time against the same number of malloc/free calls. Print the ratio.

Read your own assembly output
Medium
Write five short C functions: a loop summing an array, a struct copy, a function with and without inline, a branch that the compiler can simplify, and a pointer dereference chain. Compile each with -O0 and -O2 and look at the output on godbolt.org. Write a comment above each function explaining what changed and why. The goal is to demystify what the compiler actually produces.

Phase milestone
Software pixel canvas — animated color fields
Concept: a living screensaver / tech demo running fully on the CPU
Using only C and your own BMP writer logic (adapted to write to a raw pixel buffer in memory), produce a 400×300 software pixel buffer that animates in real-time. No game loop library — manually use clock() or QueryPerformanceCounter to measure delta time. Draw at least three animated effects per frame: a sine-wave color gradient that shifts over time using a float t that increments each frame, a moving circle drawn by testing each pixel's distance from a center point that travels along a figure-eight path, and a "plasma" effect using nested sine functions of pixel x, y, and t. Output one BMP file per frame (frame_0001.bmp, frame_0002.bmp...) for the first 60 frames, then measure how long each frame took to produce using your arena allocator for any per-frame scratch data. Print the average frame time in milliseconds.
Must implement
  • Pixel buffer as a flat uint8_t array (R, G, B packed)
  • High-resolution frame timer using OS clock API
  • Sine/cosine math for animation without math.h (use a lookup table)
  • Arena allocator for per-frame scratch memory
  • BMP output with correct 54-byte header, zero padding
Success criteria
  • 60 BMP frames produced, each visually different
  • Average frame time printed to console
  • Code under 300 lines, zero libraries
  • No memory leaks (arena reset each frame)
  • You can explain every byte in the BMP header

AABB collision detection

Phase milestone
Breakout — a complete, polished arcade game
Concept: classic brick-breaker with lives, score, multiple levels, game states
Build a complete Breakout game where the player controls a paddle to bounce a ball into bricks. This is not a prototype — it is a finished game with a clear start, play, and end. Every system must be wired together correctly: input drives the paddle, physics moves the ball, collision detection breaks bricks and bounces correctly off every surface, and game state transitions work cleanly. A random stranger should be able to pick it up and play without instructions.
Gameplay systems
  • Paddle responds to left/right keys with smooth acceleration and deceleration, not instant velocity
  • Ball bounces off paddle, walls, and bricks with correct angle reflection using the surface normal
  • Bricks take 1 or 2 hits to break depending on type — color-coded visually
  • Ball speeds up slightly every 5 bricks broken
  • 3 lives — ball lost when it falls off screen, game over when lives reach 0
  • Level ends when all bricks are cleared — next level loads with a new layout and faster starting speed
Technical requirements
  • Game states: title screen, countdown (3-2-1-GO), playing, paused (Esc), level clear, game over — all rendered differently
  • Score displayed with digit sprites or software-drawn text, increases per brick type broken
  • High score persisted to a binary file, loaded at startup
  • All rendering is software (your pixel buffer), no SDL_RenderDraw calls
  • Collision resolution uses swept AABB — no tunneling at any speed
  • Stable 60 fps measured with your frame timer, printed on screen in debug mode
Code quality
  • Under 1,500 lines of C total
  • Game state is a plain struct — no global variables
  • Asset loading separated from game logic (load function vs update function)
  • Zero dynamic allocation during gameplay — all state pre-allocated at level load

1
Triangle rasterizer with depth buffer
Medium
Implement a scanline triangle rasterizer entirely in 2D first: given three 2D vertices and a color, fill all pixels inside the triangle correctly including the flat-top and flat-bottom sub-cases. Add a float depth buffer (same resolution as framebuffer, initialized to 1.0). For each pixel, interpolate a Z value linearly across the triangle and only write the pixel if its Z is less than the current depth buffer value. Test by rendering 500 randomly positioned triangles and confirming correct depth ordering.

3D perspective transform pipeline
Medium
Implement a complete vertex transform pipeline in C: a 4×4 matrix library (multiply, transpose, inverse), and functions that construct a model matrix (translation, rotation, scale), a view matrix from eye/target/up vectors, and a perspective projection matrix from FOV, aspect, near, and far. Chain them: model × view × projection, transform a list of 3D vertices, divide by W to get NDC, then map to screen space. Visualize a spinning wireframe cube to verify correctness.

3
Perspective-correct texture mapping
Hard
Add UV coordinates to your triangle vertices. First implement incorrect affine texture mapping (linear UV interpolation in screen space) and save a screenshot. Then implement correct perspective-division: divide UV by W at each vertex, interpolate the divided values linearly across the triangle, then multiply back by interpolated 1/W at each pixel. Load a checkerboard texture and render a large oblique quad. The visual difference between the two approaches on a diagonal quad is dramatic and immediately demonstrates why the extra math matters.
Perspective-correct interpolationUV mappingAffine vs perspective error
4
Wolfenstein-style raycaster
Medium
Build a classic grid-based raycaster: a 2D tile map where 1 = wall, 0 = floor. For each screen column, cast a ray from the player position in the view direction, find the wall hit using DDA (Digital Differential Analysis), compute the wall height from the distance, and draw a vertical column of pixels. Use a simple color ramp (darker = farther) for walls. Add a 2D minimap overlay in the corner. Support free WASD movement and mouse-look for rotation. Target 60 fps at 640×400.
DDA ray castingColumn-based renderingFisheye correction2D minimap

OBJ loader and flat-shaded renderer
Medium
Write a minimal OBJ file parser that reads vertex positions and face indices. Load a free low-poly mesh (a monkey head, a teapot, or a skull). Compute the face normal for each triangle using the cross product of two edges. Shade each face flat using a single directional light: dot product of face normal with light direction gives the brightness. Backface cull faces pointing away from the camera. Render the model spinning. Aim for zero flickering — correct depth buffer usage is mandatory.

Phase milestone
Corridor crawler — a textured first-person dungeon
Concept: Wolfenstein 3D / early Doom — first-person, textured walls, enemies, an exit
Build a first-person dungeon crawler using your raycaster. The player navigates a hand-designed tile map, finds a key, and reaches an exit. There are 3 enemy types that patrol, chase the player when in range, and deal damage on contact. The goal is a complete, winnable, loseable 3-minute experience that demonstrates your full rendering pipeline working together in a real game context.
Rendering
  • Textured walls using your perspective-correct UV mapper — each wall tile uses a different texture region from a 256×256 atlas
  • Floor and ceiling rendered as horizontal spans using distance-based texture sampling
  • Sprites (enemies, items, doors) rendered as billboards sorted back-to-front and depth-tested against the wall buffer
  • Distance fog: walls and sprites darken linearly with distance (colormap lookup table, Doom-style)
  • Weapon bob animation in the lower-center of the screen — a sprite that sways with player movement speed
Gameplay
  • Player: WASD movement, mouse horizontal look, one attack action (keyboard or mouse button)
  • 3 enemy types: patroller (fixed path), chaser (pathfinds to player when visible), shooter (fires a slow projectile)
  • Health bar and collected-key indicator rendered as HUD sprites on the framebuffer
  • Door tiles that open when the player has the key and presses use (E key)
  • Game over when health reaches 0, win screen when player reaches exit tile
Technical bar
  • 60 fps at 640×400 on a 2015-era machine
  • All rendering is your software rasterizer — no GPU draw calls
  • Enemy AI uses simple BFS or A* on the tile grid — not random wandering
  • Sprites correctly occlude and are occluded by walls using the wall distance buffer
  • Level defined in a plain text file (1=wall, 0=floor, 2=enemy, 3=exit etc.)

PCM audio mixer
Medium
Use miniaudio in callback mode — the library calls your function every ~10ms requesting N samples. In that callback, mix up to 32 active sound "voices" by summing their PCM samples into the output buffer with clamping. Each voice has: a raw PCM buffer, current playback position, volume (0.0–1.0), pitch (implemented as sample rate scaling), and a loop flag. Thread pool + work queue
Hard
Build a thread pool: N worker threads (start with hardware_concurrency − 1) all blocking on a semaphore. The main thread pushes "jobs" (a function pointer + void* data) into a lock-protected queue, then signals the semaphore. Workers wake, pop a job, execute it, and decrement a pending counter. A thread_pool_wait function blocks the main thread until the pending counter reaches zero. Test by splitting a 4000×4000 pixel fill operation into 16 horizontal strips processed in parallel — measure vs single-threaded.
Thread poolSemaphore signalingJob queueParallel speedup measurement

Tile-based multithreaded rasterizer

Frustum culling system
Medium
Extract the 6 frustum planes from the camera's view-projection matrix using the standard plane extraction formula. Write an AABB-vs-frustum test: for each of the 6 planes, find the "positive vertex" of the AABB (the corner most in the direction of the plane normal) and test if it's behind the plane. If any plane rejects the AABB, the object is culled. Add this as a system that runs before rasterization. Create a scene with 1,000 objects where 90% are behind the camera — verify that culling reduces rasterization work to near-zero in that case.

Rigid body physics engine
Hard
Implement a 2D rigid body physics engine from scratch: bodies have position, velocity, angular velocity, mass, moment of inertia, and a collider shape (AABB or circle). Simulate a stack of 20 boxes falling and settling. Correctness is more important than performance at this stage.

Fiber-based job scheduler
Hard
Implement a fiber job scheduler (lightweight cooperative coroutines for job switching): create N fibers per thread using platform fiber APIs (CreateFiber on Windows, makecontext on POSIX). Jobs are C functions pushed into a lock-free MPSC queue. When a job calls job_wait(counter), the fiber yields back to the scheduler fiber and picks up a new job instead of blocking the thread. Implement a JobCounter that reaches zero when all child jobs complete, waking the waiting fiber. Test by splitting a ray-casting scene into one job per screen row — verify all rows complete before the frame advances.

Phase milestone
Design a full game level without touching code
Concept: the editor as proof — a non-programmer could build a playable level in 30 minutes
The milestone is not a specific game — it is a proof that your editor tooling is complete enough for actual level design iteration. Build a level for your Phase 3 RPG prototype entirely using the editor: tile map editor, entity placement, property inspector to configure enemies and items. Use the console to tweak values, the asset browser to swap textures, and the profiler overlay to confirm performance. Then hand the editor to someone who has never seen the codebase and ask them to build a small room with enemies and an exit.
Editor completeness
  • Tile map editor: paint, erase, pan, zoom, undo last 20 actions (circular undo buffer)
  • Entity placement: place any entity type, see its AABB in editor, delete it
  • Inspector: edit all reflected component fields on any placed entity
  • Console: 8+ working commands including timescale and spawn
  • Profiler overlay: frame time, zone breakdown, FPS counter
Workflow test
  • Design-to-playtest cycle: open editor → paint level → place entities → press F1 to play → test → F1 back to edit → adjust → repeat — all without restarting
  • Level saved and reloaded correctly across restarts
  • A second person can use the editor with no verbal instruction beyond "F1 toggles editor"
  • Any crash or data loss during the test is a failure — the editor must be stable

Emscripten WASM port
Hard
Compile your Phase 1 Breakout game to WebAssembly using Emscripten. The main loop must use emscripten_set_main_loop instead of a blocking while loop. The framebuffer blit must use SDL2's Emscripten backend or a canvas 2D context. Audio must work (use the Emscripten SDL2 audio backend). The game must load in under 3 seconds on a standard connection and run at 60 fps in Chrome and Firefox. All assets must be embedded using Emscripten's virtual file system. Measure WASM binary size and document what flags reduce it most.

2
Platform abstraction layer
Hard
Extract all OS-specific code into a platform layer interface — a single header declaring: platform_window_create, platform_blit, platform_get_input, platform_get_time_ns, platform_file_read, platform_thread_create, platform_mutex_*, platform_semaphore_*. Implement this interface for Windows (Win32 + WASAPI or SDL2), Linux (X11 or SDL2 + ALSA), and WASM (SDL2 Emscripten). Build all three with a single CMake command. Every function above the platform layer is now identical across all three targets — verify with a CI matrix.

3
CI/CD build matrix
Medium
Set up a GitHub Actions workflow that builds your engine on Windows (MSVC + Clang), Linux (GCC + Clang), macOS (Clang), and WASM (Emscripten) on every push to main. Each build must pass with zero warnings at -Wall -Wextra. Add a test step that runs a headless mode (no window, render 100 frames to a buffer, compare a checksum against a reference) to catch rendering regressions. Badge the README with build status. The goal is that every platform breaks immediately if you accidentally introduce a portability bug.

4
LTO + strip + size reduction
Medium
Take your Phase 1 Breakout game and minimize the shipping executable size: enable Link-Time Optimization (-flto), strip debug symbols (strip), use -Os optimization for size over speed, disable exceptions and RTTI (-fno-exceptions -fno-rtti), replace all CRT usage with direct syscalls for the simplest functions. Measure binary size at each step and document the contribution of each optimization. Target: under 500KB for the Windows executable, under 200KB for the WASM binary. Use a tool like bloaty or nm --size-sort to see what occupies the binary.

5
macOS Metal framebuffer blit
Hard
Implement a macOS-specific platform backend that opens a Metal-backed NSWindow, uploads the CPU framebuffer as a Metal texture each frame, and draws it as a fullscreen quad using a trivial fragment shader that just samples the texture. This replaces SDL2's software blit on macOS with a Metal blit — the GPU only does the final display upscale, all rendering is still on CPU. Measure: blit time (Metal texture upload + draw) vs SDL2 software blit at 1920×1080 output from a 640×360 source. Metal should win significantly on Apple Silicon.

Phase milestone
Ship a real game — an itch.io release that plays in the browser
Concept: the engine's ultimate test — a playable, shareable, documented game release
Take your best game from a previous phase (Phase 2 dungeon crawler or Phase 3 RPG) and release it publicly on itch.io. The WASM build plays directly in the browser — no download required. The Windows and Linux builds are downloadable. Write a short technical devlog (500 words) explaining that the game runs on a CPU-first software-rendered engine, what that means, and what hardware it targets. This is your public portfolio entry — it must be stable, complete, and documented.
Release requirements
  • WASM build plays in Chrome and Firefox with no install — load time under 5 seconds on average connection
  • Windows .exe and Linux binary downloadable from itch.io page
  • Game has a title screen, a clear win/lose condition, and working audio
  • All three builds built from CI — the release is automated, not manual
  • Executable sizes: Windows under 1 MB, WASM under 500 KB
Documentation & portfolio
  • GitHub repository with a clear README: what the engine is, how to build it, what it targets
  • Technical devlog post on itch.io page explaining the CPU-first renderer
  • One screenshot of the profiler overlay showing stable frame times
  • A "building this engine" blog post or video — the educational journey is part of the project
